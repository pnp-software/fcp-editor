FCP Profile Editor Quick Guide

The FCP Profile Editor is a web-based tool to create, edit, save and share Flight Control Procedures (FCPs) for PUS-based applications. A FCP is a conditional sequence of telecommands and telemetry checks.

FCPs are represented as UML activity diagrams which are created by dragging and dropping pre-defined elements on a canvas. The editor is customized with a mySQL database which defines the PUS telecommands and telemetry packets in SCOS-2000 format. The editor offers auto-completion facilities which allow users to quickly reference telecommands and telemetry packets and their internal parameters.

FCP models can be shared (either in read-only or in read-write mode) with other users. They are serialized to a json format which can be exported and which users can use for further processing (e.g. to generate a test sequence implementing an FCP).

Although the FCP Editor was developed with Flight Control Procedures in mind, it can be used to model any kind of control procedure for a PUS application. For instance, it could be used to define a test procedure or any other procedure consisting of sequences of telecommands and telemetry checks.

The FCP Editor is a specification-level editor. It is agnostic about FCPs are implemented. Users would typically proceed as follows:

  • The FCPs are defined in the editor
  • The FCP models (representing the formal specification of the FCPs) are exported in json format
  • The json models are processed to generate the implementation of the FCPs

The FCP Editor is based on the FW Profile Editor. Users are referred to the online help of the FW Profile Editor for information about how to edit diagrams (but the interface of the editor is very intuitive and most users will be able to find out what they need by trail-and-error). This online help focuses on the FCP-specific features of the FCP Editor.

Model Attributes

The attributes of an FCP are defined in its "Global Properties" table. Users can define for each FCP:

  • Name
  • Title
  • Project Name (useful if the editor is used for multiple project)
  • Pre-Conditions
  • Post-Conditions
  • One or more input parameters
  • Generic text notes (e.g. description of the FCP)
  • Version of the SCOS Database on which the model is built

SCOS Database

The SCOS Database must be installed on the mySQL server where the FCP Editor models reside. Its name must start with "SCOS_". There may be more than SCOS databases installed at any given time, either because the editor is used for multiple applications with different TM/TC interfaces, or because the editor is used with multiple versions of the TM/TC interface.

If no SCOS Database in installed, the editor can be used as a generic editor for UML Activity Diagrams.

The SCOS database selection is an attribute of each model. It appears in a pull-down menu at the bottom of the model's "Global Properties" table. Like all other model attributes, the database selection is also stored in the model's json serialization.

SCOS Database Loading

The SCOS database can be uploaded either as an *.sql file or as a zip file which holds the SCOS tables in csv format. The upload can only be done by the "example" user. The "example" user is configured in the configuration file config_user.php. The "example" user uploads the SCOS database by logging at: http://www.yourdomain.com/fcp-editor/users.php. Auto-Completion Facility

One important aid in the generation of FCPs is the auto-completion feature of the editor. Auto-completion is supported for the following PUS design items:

  • Telecommand packets
  • Telemetry packets
  • Telecommand parameters
  • Telemetry packet parameters
  • Event identifiers
  • Housekeeping parameters

The auto-completion feature is continuously active and as soon as it detects a string which matches one of its entries, it displays a table with the PUS information about the matching entries. Customization

The FCP Editor back-end is implemented in PHP and the front-end is implemented in javascript. The editor is available on an open and public licence and users are allowed and encouraged to extend it for their own purposes. Typical extensions might include:

  • Extension of auto-completion mechanism to handle other kinds of items (e.g. service 1 failure codes or calibration curves for TC parameters)
  • Implementation of model check function (e.g. to verify consistency with a SCOS Database version or to enforce project specific conventions on the definition of PUS items)
